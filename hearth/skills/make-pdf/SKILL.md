# Make a PDF

You have the user's content and need to produce a real `.pdf` file at a
sensible path. Use the bundled `scripts/build_pdf.py` helper — it wraps
`reportlab` (already a Hearth dep) so you don't have to write the
boilerplate every time.

## Steps

1. **Write the source markdown.** Use `write_file` to dump the content
   to `<workspace>/generated/<slug>.md`. Slug from a short user-asked
   filename or the document title.
2. **Run the builder.** Call `run_command` with:
   ```
   python <hearth>/skills/make-pdf/scripts/build_pdf.py \
     --md <workspace>/generated/<slug>.md \
     --out <workspace>/generated/<slug>.pdf \
     --title "<doc title>"
   ```
   (The exact `<hearth>` path is wherever your install lives; the
   skill loader returns the absolute path in the `folder` field —
   use that.)
3. **Confirm the result.** The script prints the final path on success.
4. **Open it for them.** Immediately call
   `run_command('powershell -Command "Invoke-Item ''<full path>''"')`
   so the PDF pops up in their default viewer. They asked for a PDF;
   they want to SEE it, not click through a path.

## Content rules — write a REAL document, not a stub
- A 1-page brief is ~250-400 words. NOT five lines of bullets.
- Use a clear title (H1), 2-4 sections (H2), and a paragraph OR 3-5
  bullets per section. No empty placeholders.
- If the topic needs research, do the research FIRST (web_search +
  web_fetch, OR spawn a researcher subagent), THEN write the doc.
- Match the user's length cue: "1-page" ≈ 350 words, "2-page" ≈ 700,
  "exec summary" ≈ 150, "deep dive" ≈ 1500+.

## NO AI-TELLS in the output PDF — this is mandatory
The user's reader should never be able to tell this was AI-generated.
The PDF is a real document for a real audience, not a Hearth demo.
**Do NOT include any of the following in the markdown source:**
- "Research compiled <date>" / "Generated by..." / "Drafted with..."
- "Hearth's take" / "Hearth itself..." / any self-reference unless the
  document is literally ABOUT Hearth.
- A "Bottom line:" auto-footer with your opinion. If a conclusion is
  needed, it's a normal H2 section ("Conclusion" / "Summary"), not a
  signoff.
- The same title repeated as title arg AND first H1 — pick one. The
  builder dedupes them now but don't rely on it; if you pass `--title
  "X"`, do NOT also write `# X` as the first line.
- Footer date / page number is handled by the builder. Don't add a
  text line for it.

The PDF should read like a human-written document. The styling +
content choices should match the topic (resume = professional tone,
academic = sober tone, pitch = bold tone) — NOT Hearth's voice.

## Use --style-file (NOT inline --style) — PowerShell mangles quotes

PowerShell shell-escaping ruins inline `--style '{"accent":...}'` —
the JSON parser sees garbage. Instead:

1. `write_file` to `<workspace>/generated/<slug>.style.json` with the
   style dict as plain JSON (UTF-8, double quotes around keys).
2. Pass `--style-file <workspace>/generated/<slug>.style.json` instead
   of `--style`.

Done. No quote hell.

## PICK A STYLE that fits the document — don't reuse the default for everything

The builder accepts `--style '<json>'`. **Choose colors + fonts that match
the TOPIC.** Don't make every PDF look like a Hearth marketing brief.

Style keys (all optional):
`accent`, `body_color`, `heading_color`, `rule_color`, `footer_color`,
`font` (Helvetica / Times-Roman / Courier), `font_bold`, `font_mono`,
`body_size`, `h1_size`, `h2_size`, `h3_size`, `leading`,
`show_rule` (bool), `show_date` (bool). Plus `--page A4|LETTER|LEGAL|A5`.

Examples to riff on — DON'T copy verbatim, vary them. Write each as a
.json file (NOT inline), then pass via `--style-file`:

- **Resume / cover letter** (write to `<slug>.style.json`):
  `{"accent":"#1f3a5f","heading_color":"#1f3a5f","font":"Times-Roman","font_bold":"Times-Bold","show_rule":false,"show_date":false}`
  Then: `--style-file <slug>.style.json --page LETTER`
- **Technical / engineering brief**:
  `{"accent":"#0b6b3a","heading_color":"#0b3d24","font":"Helvetica","body_size":10,"leading":14}`
- **Pitch / marketing one-pager**:
  `{"accent":"#c4451c","heading_color":"#c4451c","h1_size":28,"h2_size":18}`
- **Recipe / lifestyle**:
  `{"accent":"#8a4f00","heading_color":"#3d2400","font":"Times-Roman","show_rule":false}`
- **Academic / scholarly**:
  `{"accent":"#222","heading_color":"#222","font":"Times-Roman","font_bold":"Times-Bold","body_size":11,"leading":17,"show_rule":false}`
- **Minimal / monochrome**:
  `{"accent":"#000","heading_color":"#000","footer_color":"#888","show_rule":false}`

Default (warm violet) is the FALLBACK when nothing else fits, NOT a
universal choice. Read the user's ask, pick the vibe, then ship it.

## Style mechanics (built into build_pdf.py)
- A4 by default (or `--page LETTER` etc.), 1-inch margins.
- Page numbers + date in the footer (toggle with `show_date`).
- Embed images referenced as `![alt](path)` if local; skip remotes.

## Don't
- Don't try to render LaTeX. If the user wants equations, fall back to
  plain text and tell them.
- Don't paginate manually — `build_pdf.py` does it.
- Don't ship a 5-line PDF and call it a "brief". That looks lazy.
